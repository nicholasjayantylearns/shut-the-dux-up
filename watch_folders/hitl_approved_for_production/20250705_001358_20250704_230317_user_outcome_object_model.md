# 🧭 User Outcome Object

## 🎯 Purpose & Strategic Role
A **User Outcome** represents the observable benefit a user enjoys as a result of engaging in key behaviors within a given user flow. It is used to **link behavioral evidence to business impact** by way of success metrics, key signals, and intended degree of change.

## 🧠 JTBD Example
> When I need to evaluate the success of a new product enhancement, I need a way to monitor specific actions that signal people are using it, so that I can measure adoption rates and project future capacity based on demand.

## 💡 Why the User Outcome Object Matters
- Links behavioral evidence to business impact through success metrics and key signals.
- Provides observable benefits that users experience from engaging in key behaviors.
- Enables measurement of progress toward business results through user-level outcomes.
- Creates the bridge between user actions and business impact.

## 📋 Schema Attributes
| Field                | Type      | Required | Description                                                                                  |
|---------------------|-----------|----------|----------------------------------------------------------------------------------------------|
| object_type         | string    | Yes      | Must be "UserOutcome"                                                                        |
| id                  | string    | Yes      | Unique identifier                                                                            |
| behavior_id         | string    | Yes      | Links to Behavior object (end_user inherited from behavior)                                 |
| result_id           | string    | Yes      | Links to Result object (provides business context and success metrics)                       |
| end_user            | string    | Yes      | User who performs the outcome (hypothetical before UserFlow link, actual after)              |
| user_flow_id        | string    | No       | Optional link to User Flow for context (one-directional relationship)                       |
| key_signals         | [string]  | No       | Inherited from UserFlow's behaviors' key signals (LLM proposes until UserFlow links)         |
| acceptance_criteria | [string]  | Yes      | Generated by LLM based on Result's target metric at origin (coaching opportunity)            |
| outcome_statement   | string    | Yes      | Generated by LLM based on Result's target metric at origin (coaching opportunity)            |
| evidence            | [string]  | Yes      | Array of Provenance object IDs                                                               |
| tags                | [string]  | No       | System-derived tags                                                                          |
| created_at          | string    | No       | Creation timestamp                                                                           |
| updated_at          | string    | No       | Last update timestamp                                                                        |

## 🔗 Structural Role & Usage Notes
- A **User Outcome** emerges as a **junction object** formed when a Behavior and Result come together.
- It only comes into being when a Result exists that defines the desired success metric and impact target.
- The Behavior provides the end_user and signals components.
- The Result provides the business context and success metrics.
- The User Flow contains behaviors with key signals that are inherited by the outcome.
- LLM proposes key signals until UserFlow links, then inherits from UserFlow's behaviors' key signals.
- LLM generates acceptance_criteria based on Result's target metric at origin (coaching opportunity).
- LLM generates outcome_statement based on Result's target metric at origin (coaching opportunity).
- Result calculates success metrics from linked UserOutcomes (one per outcome, scaled in aggregate).
- User Outcome validity requires a valid behavior_id, result_id, acceptance_criteria, and properly formatted outcome_statement.
- Governance validates the structure and format but does not validate relationship quality or LLM inference quality.
